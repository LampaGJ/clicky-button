# Clicky Button Generator — Red-Team Report (2026-07-14)

**What this answers:** where the generator and its exported buttons break down technically and visually, what to fix first, and decided answers to eight client questions (customization gaps, realism, cross-platform, no-JS, packaging, parallelograms, per-corner styling, Tailwind).

Tracking issue: https://github.com/LampaGJ/clicky-button/issues/4

Method: an adversarial Rose/Bud/Thorn session (LUMA-catalogued evaluation method). Six real specialist agents tore the repo apart in blind parallel (divergence), then findings were scored by cross-panel corroboration and severity, clustered, and decided (convergence). Every claim below traces to a panelist finding grounded in `path:line` code evidence.

---

## TL;DR

- **The product's central promise is false as shipped.** README (README.md:3) claims "no JavaScript required in the buttons themselves: the press/hover/toggle motion is pure CSS." In the exported artifact, click-mode buttons **do not visibly press at all** without JS that the export never ships, and toggle buttons **snap instantly** with no animation. This was independently found by five of six panelists and confirmed first-hand in the exporter (app.js:855-899 writes only `.html` + `.css`, no `<script>`; the pressed look lives only inside a `.clicky-press`/`.toggle-did-interact` class added by generator-only JS at app.js:143-157 / app.js:134-137). Fix is cheap: a pure-CSS `:active` rule (the needed helper, `pressedStatePropsCss`, is already written and exported but never called — lib/clicky-button.js:654-666).
- **The engine architecture is genuinely good and the extraction succeeded.** The module is DOM-free and pure (zero `document`/`window` refs), the app imports all producers from `lib/` rather than duplicating them, geometry self-clamps against bad slider input, motion is transform-only (no layout thrash), and the two-wall physical model is a real skeuomorphic strength. These are Roses to protect through any refactor.
- **Next tier of real defects:** no `@supports`/progressive-enhancement despite modern-only CSS with hard iOS version floors; `opts.scope` only scopes the variables, not the rules or keyframes, so two styles on one page collide (blocks the packaging story); the color math emits `NaN` on any non-hex color — and the spec's own acceptance example (`faceColor: 'var(--gold)'`) is the crash case.
- **Decided calls:** Packaging → **copy-paste / relative-import registry now, npm later** (after scope + validation land). Tailwind → **No** (unanimous across three panelists). Corner-bleed approach → **sound geometry, structurally fragile** — needs one transform source of truth to stop the whack-a-mole.

## Prioritized shortlist (the converged decision)

Ordered by dot-vote (cross-panel corroboration count) × severity. Effort is rough engineering size. Rationale traces to the convergence; dissent is kept visible.

**1. Ship a pure-CSS `:active` press path, and make the toggle transition the default — so exports honor the "no-JS" claim.**
Severity/impact: CRITICAL. The flagship export is inert on click and un-animated on toggle across every platform. Corroboration: 6/6 panelists (iOS T1, illusion Q4, configurator via parity, design-critic T1/T2, code-reviewer #2, stacking-context #1) plus first-hand exporter read.
Effort: LOW–MEDIUM. `pressedStatePropsCss` already exists unused (lib/clicky-button.js:654-666); add `.clicky-btn:active .btn-face { transform/box-shadow/background-color = pressed }` and replace the toggle's `.toggle-did-interact` gate with unconditional `transition:` on the checked/unchecked face rules (today's `transition: none` at lib/clicky-button.js:601 is what makes it snap).
Rationale: highest-corroborated, lowest-cost, fixes the one claim the whole product rests on.
Dissent (kept visible): one feature genuinely cannot go pure-CSS — a full symmetric press-and-release bounce that always completes even on a tap faster than `--press-duration` (that is the legitimate reason the code reached for `animation-fill-mode: forwards` + a JS class). Resolution: keep the JS as an **optional** ~15-line enhancer shipped in the export zip and documented as progressive enhancement, never as a requirement.

**2. Add `@supports` fallbacks and fix the accessibility-critical `color-mix()` focus default.**
Severity/impact: HIGH. Container-query units (`cqi`/`cqb`) drive the entire depth illusion and font sizing (lib/clicky-button.js:347, 260-274) but do not exist before iOS Safari 16; `color-mix()` (the default `focusStyle: 'tint'`, lib/clicky-button.js:673-675) does not exist before iOS 16.2 — so on older WebKit the recessed-press shadow vanishes and the focus indicator disappears entirely (WCAG 2.4.7 failure on the default config). Corroboration: iOS (T2, T3), illusion (T5), stacking-context (context).
Effort: MEDIUM. Wrap cq-unit rules in `@supports (width:1cqi)` with a `min(8vw, …)` fallback; emit a plain `background-color` before the `color-mix` line (last-valid-wins). Also add `-webkit-tap-highlight-color: transparent` and `-webkit-touch-callout: none` to the shared face props so they ship in every export.
Rationale: turns silent disappearance into graceful degradation and closes a WCAG failure on the most common configuration.

**3. Make `opts.scope` actually scope everything (rules + keyframes), unlocking N styles per page.**
Severity/impact: HIGH. `buildCss` scopes only the custom-property block, then emits `.btn-grid`/`.btn-housing`/`.clicky-*` and `@keyframes` globally (lib/clicky-button.js:437-444). Two differently-configured buttons on one page collide last-writer-wins — and the spec's own headline use case (`{ scope: '.contact-cta-host' }`) silently fails. Corroboration: design-critic (T3), code-reviewer (Q5). This is a hard prerequisite for the packaging recommendation.
Effort: MEDIUM. Nest all selectors under the scope; suffix keyframe names with a scope hash.
Rationale: without it there is no real per-instance theming, so the "specify buttons on the fly in other projects" goal cannot be met.

**4. Harden color handling — pass through non-hex colors and guard against `NaN`.**
Severity/impact: HIGH (correctness + credibility). `darkenColor`/`hexToRgba` (lib/clicky-button.js:199-217) blindly `parseInt` hex slices; any named color, `rgb()`, 4/8-digit hex, or CSS var produces `#NaNNaNNaN` / `rgba(NaN,…)`. The module spec's acceptance criterion literally demonstrates `buildClickyCss({ faceColor: 'var(--gold)' })` — the crash case. Corroboration: design-critic (T4), code-reviewer (#9).
Effort: LOW. Detect non-hex and skip JS darkening (or switch to CSS `color-mix()` for the pressed/toggled derivation — precedent already exists at lib/clicky-button.js:674).
Rationale: cheap fix to a defect that breaks the library's own documented integration example.

**5. Collapse wall/cap transform-sync to one source of truth — end the corner-bleed whack-a-mole.**
Severity/impact: MEDIUM–HIGH (structural). Corner-bleed was fixed three times (git log 671bfec → bcc96e6 → 3143cef). Today's geometry is correct (congruent `border-radius: var(--radius)` on wall and cap, lib/clicky-button.js:464, 618), but the sync surface is still 3+ parallel places (click keyframe shared — good; toggle duplicates transforms across `toggle-press-*` vs `toggle-xform-*`; the state-test panel duplicates a third time in styles.css:186-246). Corroboration: stacking-context (#3), and directly answers the client's "sound or whack-a-mole?" question.
Effort: MEDIUM. One shared `--keycap-y` custom property both elements read, or wrap `.btn-wall` + `.btn-face` in an unstyled parent that owns the single transform.
Rationale: eliminates the bug *class* by construction instead of by vigilance.

**6. Configurator accessibility + parity pass.**
Severity/impact: MEDIUM (WCAG failures + a documented-but-missing feature). Six selects have no accessible name (rendered as `<div>` not `<label>`, e.g. index.html:182-213); dependency rows only dim via opacity so they stay keyboard-editable when "off" (app.js:223-228); tertiary text measures ≈3.10:1 contrast on the app background (styles.css:13 on styles.css:8), below the 4.5:1 AA floor; and the README promises a "Reset to defaults" (README.md:25) that does not exist anywhere in the code. Corroboration: configurator (T1, T2, T3, T5), design-critic (T14), code-reviewer.
Effort: LOW–MEDIUM. Swap the six `<div>` rows to `<label>`; set the real `disabled` property in `depRow`; bump the tertiary token or its usages to pass AA; ship the Reset button (or delete the README claim).
Rationale: multiple concrete accessibility failures plus a broken documentation promise, all low-cost.

**7. Zero-JS realism upgrade pack.**
Severity/impact: MEDIUM (product differentiation, not a defect). The illusion is a flat overhead-lit card: no specular hotspot (only a flat rim strip, lib/clicky-button.js:270-274), zero SVG filters, zero `radial-gradient`, a single fixed vertical light (every shadow is `inset 0 …`, X always 0), a shape-blind wall/cavity vignette, and an icon that never darkens on press. Corroboration: illusion (Thorns 3, 7, 8, 9, 10; Buds 1-4).
Effort: MEDIUM. Highest-leverage, all zero-JS: (a) radial-gradient specular hotspot driven by a new `--light-angle-x/-y` pair, blended `soft-light` over the face; (b) conic-gradient corner bevel replacing the four straight inset lines (app.js:535-538); (c) a second tight contact-shadow layer alongside the ambient one (lib/clicky-button.js:246); (d) derive icon color from `darkenColor()` so it participates in press-darken.
Dissent: material presets (rubber/plastic/metal/glass) are mostly config bundles over existing sliders — cheaper than they look; the `feTurbulence`/`feDisplacementMap` grain filter is the highest-effort item and should be deferred behind `@supports (filter:url(#x))`.
Rationale: this is the "make it more skeuomorphic" ask; sequence by cost, specular hotspot first.

**8. Add `package.json` + Vitest snapshot tests + Zod validation on public entry points; sync the typedef; demote the stale spec.**
Severity/impact: MEDIUM (guardrails + owner discipline). No tests, no `package.json`, no validation despite the config being the trust boundary (the owner's standing rule is a Zod schema + registry on every schema/contract). The spec's own acceptance criteria (byte-identical CSS snapshots, Node-load smoke test) are verified only by eyeballing. The `@typedef ClickyConfig` is missing ~15 real keys (all `icon*`, `useToggleColor`, `pressDarken`, `frameBevelWidth`, `textWrap`, `shadowAnimDelay`, `colorAnimDelay` — lib/clicky-button.js:97-195 vs the typedef at :23-96), and the module spec lists a ghost key `channelHeight` while citing dead pre-refactor paths. Corroboration: code-reviewer (#3, #7), design-critic (T6, T7), plus doc-rot found first-hand (comments at lib/clicky-button.js:576-581 and 619-623 still describe the wall as a "drop-shadow copy," which the real `.btn-wall` element at :453 superseded).
Effort: MEDIUM. Minimal `package.json`, 4-5 snapshot configs, a Zod validator on `buildClickyCss`/`buildClickyHtml`, mechanical typedef sync, move the spec to `claudedocs/past/`.
Rationale: closes the gap between the spec's promises and what is enforced, and aligns with the project's own schema discipline.

## Client-question answers

### Q1 — What customizations are we missing?

Grouped by where the gap bites, highest-value first:

- **Interaction/state gaps a real integrator hits in the first hour:** no `:active` hold-press (the button does not press without JS — see shortlist #1); no `:disabled` state anywhere in the engine; no `fontFamily` control (the face is hardcoded `font-family: inherit`, lib/clicky-button.js:630) — arguably the single most-wanted typographic knob and it is absent.
- **Per-button variance within a grid:** labels are per-button (comma-split, lib/clicky-button.js:923-931) but every visual property — including all icon settings — is global (lib/clicky-button.js:187-194). "REC is red, PLAY is green, each with its own icon" is an architectural ceiling, not a missing slider. The `label-${slug}` classes already exist as override hooks; a `variants: { rec: { faceColor: '#d33' } }` config key emitting scoped var overrides would deliver it.
- **Illusion knobs a serious skeuomorphic system expects:** directional light-angle; a specular/gloss primitive; SVG-filter texture (brushed metal, frosted glass, rubber grain); material presets (rubber dome / plastic chiclet / metal / glass) — the README (README.md:51) claims this range is reachable but ships only raw sliders, no packaged presets; a second contact shadow; icon-lighting integration.
- **Configurator-surfaced gaps:** the documented-but-missing Reset (README.md:25); no raw-JSON config export/import for sharing or diffing a ~70-key config (only named localStorage or full ZIP); no undo/randomize across ~70 tunable values.
- **Other absences:** text-transform, hover-darken (only hover-lift exists), intrinsic/content-driven sizing (width/height are fixed px; `max-width:100%` shrinks width but not height, so squeezed buttons squash), gradient/textured faces, circular/icon-only buttons (blocked by the missing aria-label path), and non-hex/theme-var colors (blocked by the `NaN` bug, shortlist #4).

**Decision:** add `:active` press, `:disabled`, `fontFamily`, and per-button variants before any new *visual* knob — those are the gaps that stop real integration.

### Q2 — How do we make the buttons more skeuomorphic and realistic?

All zero-JS, prioritized by realism-per-hour (this is shortlist #7):

1. **Radial-gradient specular hotspot** on the face, positioned by a new `--light-angle` pair, blended `background-blend-mode: soft-light` over `--face-color`. Cheapest glass/glossy-plastic win; also the first step toward material differentiation.
2. **Conic-gradient corner bevel** replacing the four discrete straight inset lines (app.js:535-538) so the highlight sweeps continuously around rounded corners instead of jumping in alpha at each corner vertex.
3. **Material presets** (rubber/plastic/metal/glass) as config bundles over existing sliders — mostly configuration work, high perceived payoff.
4. **Second tight contact-shadow layer** alongside the existing soft ambient shadow (lib/clicky-button.js:246) so the housing reads as *seated*, not floated.
5. **Icon press-integration:** derive `--icon-color` from `darkenColor()` so the glyph darkens with the face on press (today it is a flat sticker, lib/clicky-button.js:360).
6. **`feTurbulence` + `feDisplacementMap` grain filter** (`filter: url(#grain)`, `@supports`-gated) for rubber/plastic micro-texture — first real SVG-filter use; highest effort, defer.
7. **True 3D wall:** the current `.btn-wall` is a flat duplicate shifted straight down (lib/clicky-button.js:453-476); rotating it in local 3D space would make the side foreshorten correctly in the 3D view.

Honesty note the panel raised: the "3D view" carries baked overhead lighting rigidly as it rotates, so the light source does not stay fixed in world space — the classic texture-on-a-card tell. Either recompute per-surface lighting (SVG `feDiffuseLighting`/`feSpecularLighting`) or frame the 3D panel honestly in the UI copy.

### Q3 — How do we ensure cross-platform compatibility?

Prioritized checklist (static WebKit review; version floors are from documented Safari release history and want on-device confirmation):

1. **Make the exported click button work with zero JS** (shortlist #1) — without it the flagship export is inert on *every* platform.
2. **Gate container-query units behind `@supports` with a px/vw fallback** (lib/clicky-button.js:347, 260-274) — below iOS 16 the font-size and press shadows are dropped wholesale, collapsing the illusion rather than degrading it.
3. **Give the `color-mix()` focus default a solid-color fallback** (lib/clicky-button.js:673-675) — below iOS 16.2 the default focus indicator vanishes entirely (accessibility regression).
4. **Ship the iOS touch reset in every export:** `-webkit-tap-highlight-color: transparent` and `-webkit-touch-callout: none` on the shared face props (lib/clicky-button.js:606-651) — otherwise iOS paints a gray flash box over the keycap on every tap and long-press raises the callout menu.
5. **Fix the generator's own iPhone usability:** control inputs are `font-size:12px` (styles.css:587, 621), which triggers iOS auto-zoom-on-focus — raise to 16px (never reach for `user-scalable=no`); use `100dvh` instead of `100vh` on the app shell (styles.css:45) so content is not clipped under the dynamic toolbar; add `touch-action:none` to the 3D stage (styles.css:311) so drag rotates instead of scrolling.
6. **Self-host the icon subset or use `display=swap`** — the export injects a Google Fonts `<link>` with `display=block` (app.js:861), so offline the button face shows the literal ligature text ("send") and the "no dependencies" claim (README.md:57) is false.
7. **Add a non-`:has()` fallback for the toggle housing shadow** (lib/clicky-button.js:543-544) to cover iOS < 15.4, mirroring the sibling-selector fallback the face rules already have.
8. **Declare a support baseline in the README** (target iOS Safari 15.4+ / Chrome 105+ with the fallbacks above; full responsive sizing needs iOS 16), then verify on a real iOS 16 + iOS 15 pair, both in-Safari and Home-Screen standalone, plus Android Chrome.

Roses to keep: `touch-action: manipulation` on the face (kills the 300ms double-tap delay without disabling pinch-zoom), `@media (hover: hover)` gating everywhere (no sticky hover on touch), the genuinely pure-CSS toggle checkbox-hack, thorough `prefers-reduced-motion` handling, `-webkit-`-prefixed `backdrop-filter`, and a zoom-preserving viewport in both shells.

### Q4 — How do we keep the exported output at little-to-no JS?

**Audit result:** JS is not required to *load* the exported buttons, but it is silently required for the actual designed *motion*, and that JS is never shipped in the export.

- **Click mode:** the entire visible press effect lives in `@keyframes clicky-transform-cycle`/`shadow`/`color` gated on `.clicky-press` (lib/clicky-button.js:735-768), a class added only by generator JS on `pointerdown` (app.js:143-157). There is no `:active` fallback rule for the face. The export (app.js:864-877) ships no `<script>`. Net: the exported click button hover-lifts and then does nothing on click.
- **Toggle mode:** the checked/unchecked *state* is genuinely pure CSS (native checkbox + `:has()`/sibling, lib/clicky-button.js:862-869) — no JS needed to change state. But the animated settle transition is gated on `.toggle-did-interact` (added only by app.js:134-137), so the export snaps instantly.

**Pure-CSS alternatives (this is shortlist #1):**
- *Toggle:* replace the `transition: none` in the checked-rest rule (lib/clicky-button.js:601) with real `transition:` declarations — CSS transitions fire automatically on state change, including the first, with zero JS. (`@starting-style` handles the rare server-rendered-initial-checked edge if needed.)
- *Click:* add `.clicky-btn:active .btn-face { transform/box-shadow/background-color = pressed }` with a `transition` (not `animation`); release reverses naturally via the existing transitions. iOS fires `:active` reliably because the face already has `cursor: pointer` (lib/clicky-button.js:632). The unused `pressedStatePropsCss` helper (lib/clicky-button.js:654-666) already produces exactly these declarations.

**The one legitimately JS-tempting feature:** a press animation that always completes its full symmetric bounce even on a tap faster than `--press-duration`. A `:active` transition only animates while the pointer is physically down. Keep this as an **optional** enhancer script (the ~15 lines already specified in the comment at lib/clicky-button.js:719-733), shipped in the export zip and documented as progressive enhancement — never a requirement. Everything else that currently pulls in JS is JS-by-omission, not by necessity.

Also flag under "little-to-no external anything": the icon feature's Google Fonts dependency (above) is not JS but is a runtime network dependency that breaks the same promise.

### Q5 — How do we package this so other projects can specify buttons on the fly?

Evaluated against the actual API (`buildClickyCss`, `buildClickyHtml`, `buildClickyVars`, `defaultClickyConfig`, `internals` — a pure, DOM-free, dependency-free ES module, purity confirmed by grep):

- **CSS-custom-property contract alone — rejected as the delivery mechanism.** The generated CSS branches on config in JS (frame on/off ternary at lib/clicky-button.js:528-539, click-vs-toggle switch at :441, focus-style literals at :670-691), so a static stylesheet + vars cannot express mode/frame/focus. It is the right *theming* layer on top of a package, not the package itself.
- **Web component — rejected as the default export.** Shadow DOM would solve scoping and could bundle the activator JS, but it makes JS mandatory at runtime, killing the pure-CSS / build-time / SSR path that is the whole differentiator. Legitimate only as an *optional* additive layer for dynamic-theming consumers.
- **shadcn-style copy-paste registry — adopt now.** This matches how the module is already consumed (README Option A: unzip and drop in `.html`+`.css`; Option B: relative ES-module import). Zero new infrastructure, no premature semver on the still-churning `internals` surface (recent commits actively rewrote wall/cavity behavior), and each consumer freezes the exact CSS version they copied.
- **npm package — the target, but premature today.** The module is ~90% ready (pure ESM, zero-dep, Node-loadable), but publishing now would freeze a large, unstable `internals` surface and a typedef already out of sync with the real config.

**Decision:** ship the **copy-paste / relative-import registry now**; keep the plain relative-import path for monorepo consumers; make the **CSS-var contract the theming layer, not the delivery**. Commit to **npm** once three prerequisites land: scope actually works (shortlist #3), color handling is safe (shortlist #4), and the typedef/tests/validation close out (shortlist #8). Add a URL-hash/config-comment permalink so an exported button can be re-opened in the generator — this also kills the current device-bound, localStorage-only style library.

### Q6 — Can we use shapes other than round rects, particularly parallelograms?

Not natively today — everything is `border-radius`; grep found zero `clip-path` in the engine. Three paths, none free (two panelists converged on the same tradeoffs):

- **`transform: skewX()`** on the cell/housing is cheapest and compositor-friendly, but `transform` is already fully occupied by `translateY(...)` at 6+ rule sites (lib/clicky-button.js:715, 736-738, 801-849; styles.css:186-246) — each would need rewriting to a combined `translateY(...) skewX(...)`, plus a counter-skew on the label and icon. It also skews the shadow blur along with the shape, which is a real physics break (a parallelogram's cast shadow would not skew its own falloff). The wall/cavity "shifted-down copy" technique survives a pure skew (it is Y-translation-invariant) but breaks for any asymmetric polygon (chevron/arrow) where a vertical shift no longer yields a matching silhouette at the seam.
- **`clip-path: polygon()`** avoids skewing content but does not compose with `border-radius`, forces a new stacking context and a compositor layer per clipped element per button (a real risk against the layer budget in a dense grid), and — critically — `box-shadow` is computed against the rectangular border-box, not the clip shape, so all the ambient/inset/rim shadows keep the old rounded-rect silhouette and poke past the slanted edges. The rim-highlight band assumes a flat horizontal top edge and would no longer hug a slanted one.
- **SVG `<path>` + `filter: drop-shadow()`** is the only fully correct route (drop-shadow follows the real alpha shape), but it means giving up native `<button>` box construction and swapping `box-shadow` for the more expensive-to-animate `filter` — a genuine rewrite, not a slider.

**Assessment:** a *pure skew* parallelogram is a plausible medium effort (accept the skewed-shadow imperfection or keep shadows static). Arbitrary polygons with correct shadows require the SVG rewrite and should be treated as a separate track, not a config toggle.

### Q7 — Can we style each corner of each button differently?

- **Per-corner radius: low risk, native.** `border-radius` already takes four values. The catch is that one `--radius`/`--radius-bot` scalar is threaded through five declaration sites that must match exactly — `.btn-housing` (lib/clicky-button.js:526), `.btn-cell` (:562), cavity `::before` (:575), `.btn-wall` (:464), `.btn-face` (:618). Going per-corner means promoting all five to identical four-value strings from one source; miss one and you reproduce the exact cavity-corner-bleed bug this repo already spent three commits fixing — which is precisely why shortlist #5 (single transform/geometry source) should land first.
- **Per-corner bevel/lighting: not expressible today.** The frame bevel is four straight-edge inset shadows that shade *edges*, not corner arcs (app.js:535-538); no current primitive can paint "top-left corner brighter than bottom-right," and neither `frameBevelAlpha` nor `frameBevelWidth` can express it. Real per-corner lighting needs new radial/conic corner overlays (or SVG `feSpotLight`/`feDiffuseLighting` per corner), correctly inserted into the fully-implicit (no z-index) paint order between cavity and cap — a materially new rendering feature.

### Q8 — Should we use Tailwind?

**Decided: No.** Unanimous across the three panelists in whose domain it fell (design-critic, code-reviewer, configurator). Rationale:

- The product's shipped artifact *is* a self-contained, dependency-free, build-step-free `.html`+`.css` pair (README.md:3, :19). Emitting Tailwind classes would make every export depend on the consumer's build pipeline and Tailwind version — directly against the value proposition and the owner's minimal-dependency preference.
- The engine's core output — computed multi-layer inset `box-shadow` stacks, config-derived cubic-bezier keyframes, `cqb`/`cqi` container-query math (lib/clicky-button.js:249-284, 735-767) — is exactly what utility classes express *worst*; it would become arbitrary-value soup (`[box-shadow:inset_0_8cqb_12cqb…]`) with zero legibility gain.
- The generator's own control panel (index.html/styles.css) is already a small, token-driven stylesheet whose actual defects are markup-semantics and token-value bugs (unlabeled selects, fake-disabled rows, a failing contrast token) — none of which Tailwind would touch; it would only add a build step to a deliberately zero-build tool.

The only Tailwind-shaped opportunity is *additive and optional*: a future **export target** that wraps the generated vars in a Tailwind preset/plugin for Tailwind-native consumers — an output format, never an internal dependency.

## Panelist teardown appendices (condensed)

All six were dispatched real library agents (no role-play, no gaps, no re-prompts). Severities are each panelist's own; full `path:line` evidence is in the shortlist and question sections above.

**CSS stacking-context & animation debugger (dispatched: agent-css-stacking-context-animation-debugger).**
Top thorns: (1 critical) click-mode press never fires in the export, `pressedStatePropsCss` written but never called; (2 high) toggle animation also JS-gated; (3 high) wall/cap transform-sync is manual across 3+ sites — the corner-bleed bug class; (4 high) up to 10 `box-shadow` layers animate main-thread per press, a paint-budget threat in a grid; (medium) zero z-index (a Rose, but any future partial z-index inverts paint order), `preserve-3d` nullified by `overflow:hidden` on the housing, `:has()` for all state matching, no `will-change`, hover timing hardcoded `100ms` ignoring the easing controls, shadow easing locked linear while transforms can overshoot (visible desync). Roses: no z-index war, `container-type` isolation, transform-only motion, shared click keyframe. Buds: single `--keycap-y` source, opacity-crossfade pre-painted shadows, scoped `will-change`.

**CSS/SVG 3D-illusion designer (dispatched: agent-css-svg-3d-illusion-designer).**
